# Правила для ИИ по написанию кода

## Цель
Этот документ задает обязательные правила для ИИ при генерации, изменении и рефакторинге кода в проекте. Цель — получать код, который легко читать, тестировать, расширять и поддерживать в долгосрочной перспективе.

ИИ обязан выбирать простые, надежные и предсказуемые решения, а не демонстрировать сложность ради сложности.

---

## Базовые принципы

### 1. KISS
- Всегда выбирай самое простое решение, которое корректно решает задачу.
- Не усложняй архитектуру без явной необходимости.
- Не вводи абстракции, паттерны, слои, классы, хуки, сервисы и утилиты заранее, если задача решается проще.
- Избегай чрезмерной вложенности, магии и неочевидного поведения.

### 2. YAGNI
- Не реализуй функциональность, которая не нужна прямо сейчас.
- Не добавляй “на будущее” дополнительные параметры, уровни конфигурации, generic-абстракции, интерфейсы, фабрики и расширения без реальной потребности.
- Не создавай код на основе гипотетических сценариев, если они не входят в текущую задачу.

### 3. DRY
- Не дублируй логику, строки, условия, вычисления и шаблоны компонентов без причины.
- При повторении выноси общую часть в функцию, хук, утилиту, компонент или сервис.
- Не злоупотребляй DRY: если абстракция ухудшает читаемость, выбирай понятность.

### 4. SOLID
Используй принципы SOLID там, где они реально улучшают код.

#### S — Single Responsibility Principle
- Каждый модуль, компонент, класс и функция должны иметь одну понятную ответственность.
- Не смешивай бизнес-логику, UI, форматирование данных, побочные эффекты и доступ к данным в одном месте.

#### O — Open/Closed Principle
- Пиши код так, чтобы его можно было расширять без переписывания базовой логики.
- Предпочитай расширяемые интерфейсы, конфигурации и композицию вместо постоянного редактирования старого кода.

#### L — Liskov Substitution Principle
- Наследование и замены типов должны сохранять ожидаемое поведение.
- Не используй наследование там, где лучше подходит композиция.

#### I — Interface Segregation Principle
- Не создавай слишком широкие интерфейсы.
- Пусть зависимости получают только то, что им реально нужно.

#### D — Dependency Inversion Principle
- Завязывайся на абстракции, а не на детали реализации, когда это действительно оправдано.
- В прикладном коде предпочитай внедрение зависимостей и отделение внешних интеграций от ядра логики.

### 5. Composition over Inheritance
- Всегда предпочитай композицию наследованию, если нет веской причины делать иначе.
- Для UI предпочитай композицию компонентов, slots, props и конфигурации.

### 6. Separation of Concerns
- Разделяй ответственность между слоями приложения.
- UI не должен содержать сложную бизнес-логику.
- Работа с API, преобразование данных, валидация, вычисления и отображение должны быть отделены друг от друга.

### 7. Readability First
- Код должен быть понятен человеку без дополнительных объяснений.
- Читаемость важнее “умности”.
- Не используй сокращения, если они ухудшают понимание.
- Имена сущностей должны ясно отражать смысл.

## Декомпозиция кода

### Обязательное правило декомпозиции
Если логика становится объемной, ИИ обязан разбивать код на несколько файлов.

Не допускается создавать один большой файл, если в нем смешаны:
- UI
- бизнес-логика
- работа с API
- типы
- константы
- утилиты
- валидация
- состояние
- анимации
- обработчики форм

### Когда нужно выносить код в отдельные файлы
Выноси код в отдельные файлы, если выполняется хотя бы одно условие:
- файл становится слишком длинным
- компонент делает слишком много вещей
- часть логики можно переиспользовать
- участок кода трудно читать без сворачивания блоков
- появляются повторяющиеся фрагменты
- есть отдельная бизнес-логика, не относящаяся напрямую к отображению
- появляются самостоятельные типы, схемы, конфиги или константы

### Рекомендуемая декомпозиция
Разделяй код на:
- `components/` — переиспользуемые UI-компоненты
- `sections/` — крупные секции страницы
- `features/` — функциональные модули
- `lib/` — утилиты, helpers, форматирование, общие функции
- `services/` — работа с API, внешними системами, запросами
- `hooks/` — пользовательские хуки
- `types/` — типы и интерфейсы
- `constants/` — константы и конфигурации
- `schemas/` — схемы валидации
- `store/` — состояние приложения, если оно есть

### Правило размера
- Функции должны быть небольшими и фокусными.
- Компоненты не должны превращаться в “бога-компонента”.
- Если компонент трудно быстро просканировать глазами, он требует декомпозиции.

## Правила проектирования

### 1. Ясная структура проекта
- Соблюдай предсказуемую структуру папок.
- Все сущности должны лежать там, где их ожидают найти.
- Не раскладывай логику хаотично.

### 2. Локальность изменений
- Новая функциональность должна требовать минимального количества изменений в старом коде.
- Старайся проектировать код так, чтобы изменение одной части не ломало остальные.

### 3. Минимальная связность
- Снижай coupling между модулями.
- Избегай скрытых зависимостей.
- Передавай зависимости явно.

### 4. Высокая связность по смыслу
- Код, который относится к одной задаче, должен быть сгруппирован вместе.
- Не размазывай одну ответственность по всему проекту.

### 5. Предсказуемость
- Не используй неожиданное поведение, скрытые сайд-эффекты и неявные изменения состояния.
- Код должен работать очевидно.

---

## Правила для функций

- Одна функция — одна задача.
- Функции должны быть короткими и понятными.
- Аргументы должны быть минимально необходимыми.
- Не передавай в функцию лишние данные.
- Избегай функций с большим числом параметров.
- По возможности используй объект параметров для сложных вызовов.
- Не смешивай вычисление результата и побочные эффекты в одной функции.
- Функции должны иметь говорящие имена: `calculateTotalPrice`, `formatScheduleLabel`, `buildNavigationItems`.

---

## Правила для компонентов

- Компонент должен отвечать за один UI-сценарий.
- Не смешивай layout, данные, анимации и тяжелую бизнес-логику в одном компоненте.
- Если компонент разрастается, выноси:
  - подкомпоненты
  - хуки
  - утилиты
  - типы
  - константы
- Props должны быть понятными и минимальными.
- Не создавай универсальные компоненты “на все случаи жизни”, если в этом нет реальной потребности.
- Предпочитай явные API компонентов.
- Избегай слишком глубокой проп-дриллинговой структуры — при необходимости используй композицию или локальный state management.

---

## Правила для типов и моделей

- Всегда описывай значимые сущности явными типами.
- Не используй `any`, кроме действительно исключительных случаев.
- Предпочитай точные типы вместо размытых.
- Типы должны отражать доменную модель, а не случайную форму данных.
- Если тип переиспользуется в нескольких местах, вынеси его отдельно.
- Не дублируй одинаковые типы под разными именами.

---

## Правила для работы с состоянием

- Держи состояние как можно ближе к месту использования.
- Не поднимай state выше без необходимости.
- Не используй глобальное состояние, если достаточно локального.
- Производные значения должны вычисляться, а не храниться отдельно, если это не дает явной выгоды.
- Избегай дублирования состояния.
- Для асинхронного состояния учитывай загрузку, успех и ошибку.

---

## Правила для работы с API и данными

- Взаимодействие с API должно быть отделено от UI.
- Не пиши сетевые запросы прямо в визуальных компонентах, если это можно вынести.
- Создавай отдельные слои для:
  - запроса
  - трансформации данных
  - обработки ошибок
- Форматы данных с сервера не должны бесконтрольно протекать во все слои приложения.
- При необходимости создавай mapper / adapter между API-моделью и UI-моделью.

---

## Правила для ошибок

- Обрабатывай ошибки явно.
- Не проглатывай исключения молча.
- Ошибки должны быть либо корректно залогированы, либо показаны пользователю в понятном виде.
- Сообщения об ошибках должны быть полезными.
- В потенциально ненадежных местах добавляй защиту от некорректных данных.

---

## Правила для комментариев

- Комментарии не должны объяснять очевидное.
- Код должен быть настолько ясным, чтобы нуждаться в минимуме комментариев.
- Комментарии нужны там, где важно объяснить причину, ограничение, компромисс или неочевидное решение.
- Не оставляй мусорные комментарии и закомментированный старый код.

---

## Правила для именования

- Используй понятные, полные и однозначные имена.
- Не используй бессмысленные названия вроде `data`, `value`, `item`, `handler`, если можно назвать точнее.
- Названия должны отражать роль сущности в системе.
- Для булевых значений используй имена, отвечающие на вопрос да/нет: `isOpen`, `hasError`, `canSubmit`.

---

## Правила для рефакторинга

- При изменении кода сначала сохрани корректность поведения.
- Убирай дублирование, упрощай ветвления, уменьшай размер функций и компонентов.
- Не проводи “архитектурную революцию” без необходимости.
- Каждый рефакторинг должен делать код проще, а не просто “иначе”.

---

## Правила для производительности

- Сначала корректность и ясность, потом оптимизация.
- Не оптимизируй преждевременно.
- Оптимизация должна быть осмысленной и локальной.
- Избегай лишних ререндеров, лишних вычислений и лишних запросов.
- Тяжелая логика должна быть вынесена из рендера.

---

## Правила для безопасности и надежности

- Не доверяй внешним данным по умолчанию.
- Валидируй входные данные там, где это важно.
- Не храни чувствительные данные в небезопасных местах.
- Не добавляй небезопасные решения ради скорости разработки.
- Учитывай null/undefined, пустые состояния, ошибки сети и некорректные ответы API.

---

## Правила для UI-кода

- Интерфейс должен быть предсказуемым и консистентным.
- Не дублируй визуальные паттерны — выноси их в переиспользуемые компоненты.
- Следи за единым подходом к spacing, typography, colors, states.
- Loading, empty state, error state должны быть предусмотрены там, где это необходимо.
- Accessibility не игнорировать: семантика, aria-атрибуты, фокус, клавиатурная навигация.

---

## Что ИИ НЕ должен делать

- Не писать огромные файлы без необходимости.
- Не смешивать несколько ответственностей в одном модуле.
- Не добавлять абстракции “на вырост”.
- Не плодить хелперы и утилиты без пользы.
- Не использовать сложные паттерны, если простое решение достаточно.
- Не дублировать код ради скорости.
- Не использовать `any` без сильной причины.
- Не оставлять мертвый код.
- Не делать неявные сайд-эффекты.
- Не ломать существующий стиль проекта без причины.

---

## Что ИИ должен делать всегда

- Сначала понимать задачу.
- Затем предлагать простую архитектуру.
- При объемной логике — декомпозировать код на несколько файлов.
- Соблюдать KISS, YAGNI, DRY, SOLID.
- Писать чистые имена и маленькие функции.
- Разделять UI, данные, логику, типы и конфигурацию.
- Думать о поддерживаемости кода через 6–12 месяцев.
- Предлагать решение, которое удобно развивать, а не только быстро сгенерировать.

---

## Формат ответа ИИ при генерации кода

При реализации задачи ИИ должен:

1. Кратко описать решение.
2. Объяснить, почему выбрана именно такая структура.
3. Показать декомпозицию по файлам.
4. Для каждого файла указать его назначение.
5. Дать чистый, согласованный код.
6. Отдельно отметить, где логика вынесена для соблюдения KISS, DRY и SRP.
7. Если есть компромиссы — явно назвать их.

---

## Итоговое правило

Если есть выбор между:
- более умным, но сложным кодом
- и более простым, читаемым и поддерживаемым кодом

всегда выбирай более простой, читаемый и поддерживаемый вариант.

Важно: не уходи в общие советы. Давай конкретные решения и сразу код. Каждый этап завершай списком файлов, которые были созданы или изменены. Если код большой, разбивай его логично. Следи, чтобы все части были совместимы друг с другом.

При принятии визуальных решений ориентируйся на премиальные fashion / performance / creative studio сайты. Интерфейс должен ощущаться дорогим, минималистичным и эмоциональным. Лучше меньше элементов, но сильнее композиция.
